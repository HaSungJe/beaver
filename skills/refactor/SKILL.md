# Beaver — Refactor (계획 기반 구조 정리 / 공통화)

동작을 바꾸지 않고 코드 구조를 개선한다. beaver의 spec-first 원칙대로 **먼저 계획서를 쓰고 → 사용자가 조정·승인 → 실행**한다(plan→build와 같은 흐름, refactor 전용 계획서 사용). 강조점은 **비슷한 기능 묶기**(중복 제거·공통 로직 추출)이며, 분리 기준은 `CLAUDE.md` 규약을 따른다.

```
green 확인 → 대상 식별 → 📄 refactor-plan 작성 → 사용자 조정·승인 → 단계별 실행+테스트 → 보고
```

## 0. green baseline 확인

`.beaver/config.json` 의 `commands.test` 를 먼저 1회 돌려 **현재 통과 상태**를 확인한다. baseline이 깨져 있으면 알리고, 그 위에서 리팩토링하면 회귀 판별이 불가함을 설명한다.

## 1. 대상 식별

사용자 지정 범위(파일/디렉터리/기능군)가 있으면 그 안에서, 없으면 다음을 스캔(Grep/Read):
- **중복/유사 로직** — 거의 같은 코드가 여러 곳에 반복되는가?
- **잘못 놓인 로직** — `CLAUDE.md` 의 "공통 로직 분리 기준"상 util/module로 빠져야 할 게 service/controller에 인라인돼 있는가?
- **비슷한 기능 군집** — 흩어진 관련 기능을 한 모듈/util로 묶을 여지.

각 발견은 **실제 코드 근거(경로:라인)** 와 함께 수집한다.

## 2. refactor-plan 작성 (계획 게이트)

`${CLAUDE_PLUGIN_ROOT}/templates/refactor-plan.md` 기반으로 `.beaver/output/refactor/<name>-refactor-<YYMMDD>.md` 작성:
- 목표/범위(+비범위), green baseline 상태
- 발견 목록(유형·위치·근거·규약 근거)
- 변경 방안(어디로 묶/추출 — 규약대로)
- **작은 단위 실행 순서**(각 단계 후 테스트)
- 영향 파일 / 동작 보존·테스트 전략 / 리스크

## 3. 사용자 조정·승인

작성한 계획서를 사용자에게 제시하고 **우선순위·범위를 조정**받는다. 한 번에 너무 넓게 가지 않는다 — 사용자가 범위를 줄이거나 항목을 빼면 계획서를 갱신한다. **승인 전에는 코드를 건드리지 않는다.**

## 4. 실행 (단계별)

승인된 계획서의 실행 순서대로, 작은 의미 단위로 진행:
- 공통 로직은 `CLAUDE.md` 규약 위치로 추출(전역 util / 도메인 util / 공용 module). 호출부를 새 추출물로 교체, 시그니처·네이밍 규약 준수, 죽은 코드/중복 정의 제거.
- **각 단계 후 테스트 재실행**해 green 유지 확인. 깨지면 즉시 처리하거나 직전 단계로 되돌린다.
- 계획서의 실행 순서 체크박스를 `[x]` 로 갱신하며 진행.

## 5. 검증·보고

- 전체 `commands.test` 통과로 동작 보존을 입증(통과 못 하면 리팩토링 미완료).
- 무엇을 어디로 묶었고 왜 그렇게 나눴는지(규약 근거 포함) 파일별로 요약 보고.
- 큰 리팩토링이면 의미 단위로 커밋을 나눌 것을 제안(`/beaver:ship`).

## 주의

- **동작 변경(기능 추가/수정)은 리팩토링이 아니다** — 그건 `/beaver:plan`→`/beaver:build` 경로로.
- 테스트가 없는 영역을 리팩토링할 땐 위험을 먼저 알리고, 필요하면 characterization 테스트를 먼저 두자고 제안(계획서에 명시).

## 관련
