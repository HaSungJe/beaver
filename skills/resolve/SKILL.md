# Beaver — Resolve (병합 충돌 해결)

merge / rebase / cherry-pick / pull 중 발생한 충돌을 안전하게 해결한다.

> **두 가지로 발동된다:** ① `/beaver:ship` 의 dam 병합 중 충돌이 나면 ship이 이 절차를 **자동으로** 이어서 수행한다. ② 그 밖의 충돌(수동 rebase/pull/cherry-pick 등)은 이 skill을 직접 호출하거나 자연어로 발동한다.

## 0. 상태 파악
1. `git status` 로 충돌 파일 목록과 현재 작업(merge/rebase/cherry-pick)을 확인한다.
2. 각 충돌 파일에서 `<<<<<<<` / `=======` / `>>>>>>>` 마커 구간을 읽고, **ours / theirs 양쪽이 무엇을 의도했는지** 파악한다(필요하면 양쪽 커밋 로그·전후 맥락 확인).

## 1. 해결안 도출
- 충돌이 **단순 수용**(한쪽이 명백히 상위)이면 그쪽 채택.
- **양쪽 변경이 모두 의미 있으면** 통합 — 두 의도를 모두 살리되 `CLAUDE.md` 규약(네이밍·구조·에러처리 등)에 맞게 정리. 임의로 한쪽을 버리지 않는다.
- 의미가 불확실하거나 비즈니스 판단이 필요하면 **사용자에게 양쪽 의도를 요약해 제시하고 선택을 받는다**. 추측으로 밀어붙이지 않는다.

## 2. 적용
1. 각 파일의 마커를 제거하고 통합 결과로 정리. 마커 잔여가 없는지 재확인(`git diff --check`).
2. 해결한 파일을 `git add`.
3. 가능하면 영향 범위 테스트를 `.beaver/config.json` 의 `commands.test` 로 돌려 회귀가 없는지 확인.

## 3. 마무리
- 사용자에게 **무엇을 어떻게 통합했는지 파일별로 요약** 보고하고 확인받는다.
- 승인되면 작업 이어가기: merge면 커밋, rebase면 `git rebase --continue`, cherry-pick이면 `--continue`.
- 해결이 위험하거나 범위가 크면 `git merge --abort` / `git rebase --abort` 로 되돌리는 선택지도 제시.

## 주의
- 충돌 해결 커밋을 사용자 확인 없이 완료하지 않는다.
- 마커를 지웠다고 끝이 아니다 — 통합 결과가 **컴파일·테스트 통과하는지**까지가 해결이다.

## 관련
- 리팩토링이 필요해지면: [refactor](../refactor/SKILL.md)
