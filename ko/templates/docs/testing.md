## Data-Access 스모크 (DB 없는 통합)

<!-- mock 경계 = 검증 경계. 단위 테스트가 data-access 레이어를 mock 하면 쿼리 매핑 통합 접점(조건→저장소 컬럼 해석, 쿼리/statement 생성, 제약 오류 매핑, 라이프사이클 훅, 연관 데이터 로딩)은 어느 spec 에서도 실행되지 않는다 — 이 클래스의 버그는 구조적으로 모든 테스트를 통과한다. 이 사각지대를 여기에 명시한다. -->
<!-- 정의(다른 단계가 이 정의를 참조): **매핑 민감 쿼리 구문** = 매핑된 필드명/형태가 실제 저장소 컬럼과 다른 필드를 쓰는 조건(예: 관계/join 필드, 개명된 컬럼), 또는 동적으로 조립되는 쿼리(예: query-builder/criteria API, 문자열 조립 조각). 이 프로젝트의 data-access 스택이 실제로 부르는 이름으로 번역한다 — 코드 근거(경로:라인)로 도출. -->
<!-- 이 프로젝트의 data-access 스택이 실제 연결 없이 쿼리/metadata 를 빌드할 수 있으면(예: TypeORM `DataSource.buildMetadatas()` + `createQueryBuilder(...).getSql()`; 다른 스택의 metamodel 빌드 / statement compile 상당물 — 이 프로젝트가 실제 쓰는 스택에서 도출), 스모크 규약을 기록:
     1) 각 data-access 메서드의 조건이 — 선언적 조건이든 동적 조립이든 — 예외 없이 쿼리/statement 문자열로 빌드되어야 함; 매핑된 필드명이 저장소 컬럼으로 해석되지 않는 곳은 여기서 즉시 fail;
     2) 스택의 metadata 에서 위험 매핑(매핑된 필드명 != 저장소 컬럼명인 필드)을 자동 추출해 목록을 스냅샷 — 새 위험 매핑이 생기면 리뷰를 강제.
     수 초 내 실행, DB/외부 서비스 불필요, CI 상시. 스택에 무연결 빌드 API 가 없으면 대안을 대신 기록: 핵심 data-access 유닛 한정 실DB 통합(testcontainers/임베디드 DB). data-access 레이어가 없는 프로젝트는 이 섹션 삭제. -->
<!-- 강제 규칙: 매핑 민감 쿼리 구문을 쓰는 data-access 메서드를 data-access mock spec 만으로 커버하는 것 금지 — 스모크(또는 실DB) 케이스 필수. -->
