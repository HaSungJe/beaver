# plan — 기능 기획 (spec → plan / revision)

**격리 옵트인.** beaver의 기본 기획 진입점은 `/beaver:fast`(워크트리 없이 현재 브랜치에서 바로)다; `plan`은 격리·병렬 안전 작업을 위해 stick 워크트리를 명시적으로 원할 때만 쓴다 — 기획 절차 자체는 동일하다.

## 0. 전제 (메인 레포 — 읽기만, 설정 시드 1개 제외)
워크트리가 생기기 전 단계라 메인 레포에서 읽는다.
- `CLAUDE.md` 필요. 없으면 중단하고 `/beaver:analyze` 안내.
- `.beaver/config.json`에서 경로·`branch` 설정을 읽는다.
- **워크트리 설정 보장**: `.claude/settings.json`의 `worktree.baseRef`는 `"head"`여야 한다 — 없거나 다르면 설정한다(merge-patch, 다른 키 보존). 그래야 EnterWorktree가 origin/기본브랜치가 아닌 현재 HEAD에서 stick을 분기한다. **기획 중 유일하게 허용되는 메인 레포 쓰기**(EnterWorktree가 워크트리 생성 전에 baseRef를 읽는다). 멱등이며, §2 이후는 전부 워크트리 로컬.

## 1. 모드 판별
대상 기능명으로:
- 같은 기능의 `plan.md` + `report.md` 존재(구현 완료) → **변경 모드**(revision).
- 그 외 → **신규 모드**(spec → plan).
애매하면 사용자 확인.

## 1.5 진행 중 타 영역 요청
진행 중 들어온 다른 영역의 요청도 기획한다 — 미루지 않고 문서를 생성하며, 그 요청도 §1 모드 판별을 따로 거친다. 두 영역이 맞닿거나 영향을 주면 작업 중이던 문서에 추가하고, 아니면 새 영역의 도메인 경로에 문서를 새로 만든다.

## 1.6 build 전까지 문서만
여기서부터 `/beaver:build` 전까지, 이 흐름의 모든 사용자 요청은 기획 요청이다 — 문서를 쓰거나 고치는 것으로 답하고, 소스·테스트 코드는 절대 편집하지 않는다. "X 추가해줘", "Y 지워줘", "Z 고쳐줘"처럼 직접 코드 작업으로 들리는 요청도 spec/plan/revision에만 반영하고 문서 변경으로 보고한다. 허용되는 쓰기: `.beaver/output/` 문서, `.beaver/memory/`, §4.5 draft 규약 문서, §0/§2 관리 파일. 문서가 항상 코드보다 최신이어야 한다 — 여기서 코드를 고치면 그게 깨진다.

## 1.7 병합·불필요 코드 확인 (코드 추가·삭제 설계 공통)
추가를 설계하기 전에, 새 코드가 합류할 기존 코드 — 파일·유닛·패턴 — 를 먼저 찾고 그 근거(경로:라인) 위에서 연계를 설계한다. 이어서 그 변경으로 불필요해지는 기존 코드가 있는지 확인한다 — 대체되는 분기, 아무도 안 부르는 헬퍼, 안 쓰는 import, 죽은 설정 — 있으면 그 삭제를 변경 목록에 포함한다. 추가만 하고 대체한 것을 지우지 않는 설계는 미완이다.

## 1.8 형제 코드 계약 준수 (신규 유닛 설계 공통)
새 함수·유닛을 설계하기 전에 형제 코드를 먼저 찾는다 — 같은 모듈이나 인접 코드에서 같은 종류의 동작을 하는 기존 유닛. 형제가 있으면 새 유닛은 형제의 책임 분할을 따른다: 에러를 어디서 처리하는지, 무엇을 throw·반환하는지, 결과 검증은 누가 하는지. 제안에 그 형제를 근거(경로:라인)로 명시한다. 형제와 다른 분할로 가는 것은 결정사항이다 — 이유와 함께 제안하고 spec/revision 확정 전에 사용자 확인을 받는다. 형제를 인용하지 않고 다른 분할을 택한 설계는 미완이다.

## 2. 워크트리 먼저 진입 (stick 격리)
**어떤 쓰기보다 먼저** 워크트리에 진입한다 — §0–§1의 git 읽기만 이보다 앞선다. 이후 모든 쓰기는 워크트리 로컬로 들어가고 ship이 stick을 머지할 때만 원본 브랜치에 도달한다. 원래 작업 디렉터리는 그대로 둔다(병렬 세션 가능).

- **이미 stick worktree 안이면**(cwd가 `.claude/worktrees/` 하위 + `.beaver/.auto-branch-state.json`에 현재 브랜치가 키로 존재) → 그대로 누적.
- 아니면:
  1. `origin_branch = git branch --show-current` — ship이 되돌릴 대상. 빈값(detached)이면 중단하고 브랜치 체크아웃 안내.
  2. 요청 이름 = `<stick_prefix>/<domain>-<rand6>` (기본 `stick/...`). 도메인은 기능명/요청에서 추출.
  3. `EnterWorktree(name=<요청 이름>)` → 워크트리 생성 + 세션 cwd 전환(base = §0 baseRef에 따라 현재 HEAD). **하니스가 이름을 정규화한다** — 디렉터리는 `.claude/worktrees/<이름의 / → +>`, 브랜치는 `worktree-<정규화된 이름>` (예: `stick/user-a1b2c3` → 브랜치 `worktree-stick+user-a1b2c3`); 요청한 이름이 그대로 남는다고 가정하지 말 것.
  4. 진입 후 **실제 생성된 값**을 읽는다: `stick = git branch --show-current`(실제 브랜치), 워크트리 경로 = cwd. `.beaver/.auto-branch-state.json`에 `{ "<stick>": "<origin_branch>" }`를 그 실제 브랜치를 키로 기록 — ship의 병합/삭제가 이 키로 브랜치를 찾으므로 추측한 이름을 쓰면 ship이 깨진다. 프로젝트 `.gitignore`에 그 라인이 없으면 추가.

  워크트리에 의존성 디렉터리는 채우지 않는다 — 코드를 실행하는 단계가 없고, 테스트는 `/beaver:test`에서만 실제 체크아웃 위에서 돈다.

생성한 worktree·stick·origin_branch를 한 줄로 알린다. 둘 다 로컬 전용 — 원격 push는 ship에서만.

**이제 워크트리 안 — memory 먼저 읽기**: `.beaver/memory/`(MEMORY.md + 관련 토픽)를 **최우선** 반영(memory > CLAUDE.md). 사용자가 지속 규칙을 지적하면 확인 후 저장(워크트리 로컬, stick과 함께 ship됨) — 프로토콜 `${CLAUDE_PLUGIN_ROOT}/templates/memory-protocol.md`.

## 3. 신규 모드 (분석 → 대화 → spec → plan)
> **결정사항**(decision) = CLAUDE.md/memory만으로 못 정해 사용자 확정이 필요한 항목.
> **산출물** — spec: `.beaver/output/spec/<domain>/<feature>-spec.md`(`templates/spec.md`), plan: `.beaver/output/plan/<domain>/<feature>-plan.md`(`templates/plan.md`).

1. **심층분석 — 작업 크기에 맞춘 fan-out.** 가벼운 인라인 사전 스캔으로 기능의 도메인 디렉터리 + 인접 서브시스템(= **읽기 범위**)을 찾고 루틴/신규를 판단한다. 서브에이전트마다 풀 컨텍스트 부팅 + 파일 콜드 읽기 비용이 들므로 이득 나는 곳에만 쓴다:
   - **기존 패턴 매칭 루틴 기능** → fan-out 없음; 찾은 경로에 인라인 스코프 패스 1회.
   - **신규/다중 서브시스템/복잡** → 병렬 fan-out(Workflow 병렬 / Task 분산 / 불가 시 순차)하되, 각 에이전트에 **명시적 읽기 범위**를 준다 — "코드베이스 전체 읽기" 금지. 해당되는 에이전트만(2~4개) 쓴다. 에이전트 `${CLAUDE_PLUGIN_ROOT}/agents/`:
     - architecture-mapper — 인접 서브시스템 구조
     - convention-scout — 해당 도메인 규약
     - test-pattern-analyzer — 테스트 규약: 이 프로젝트의 실제 테스트 구성을 코드 근거(경로:라인)로 도출, 프로젝트의 도구·이름 그대로. 특정 스택 구문을 가정하지 않는다.
   - 재사용/인접 스캔(**항상 인라인**, 에이전트 아님) — 유사 기능·재사용 단위·영향받는 데이터/상태 전부 코드 근거(경로:라인)로, 프로젝트가 쓰는 이름 그대로 도출.
2. **판별** — ① **기존 패턴에 추가**(이 프로젝트 코드가 실제로 하는 것에 매칭되는 루틴 기능) 또는 ② **신규(net-new)**(현 코드/규약에 없는 영역). 신규면 이 프로젝트가 실제로 정해야 하는 결정포인트에 **구현 기술검토 + 제안**(필요 라이브러리/접근/대안)을 더한다.
3. **제안** — "기존 `X` 연계 / 기존 `Y` 패턴 재사용 / 설계 접근 2-3안(tradeoff·권장)"을 근거(경로:라인)와 함께.
4. **대화형 1문1답** — spec을 빈칸으로 던지지 않고 설계 접근안 선택·미정 결정사항을 **한 번에 하나씩** 묻고 답을 수집한다.
5. **spec 자동생성** — 확정되면 spec 문서를 자동 작성(기능 설명 / API / 비즈규칙 / 참고 + 코드근거 제안 + 확정된 결정사항·근거). 미답 결정사항 없어야 다음 단계.
6. **plan 작성** — plan 문서: 파일 목록 / 레이어별 설계 / 테스트 케이스 / 응답 코드 + **사전 구현 필요 항목**(인프라 부재 시 `- [ ]`, 모두 `[x]` 전엔 build 불가). **해당 시 data-access 스모크 필수**: 계획된 data-access 메서드가 매핑 민감 쿼리 구문(정의: docs/testing.md "Data-Access 스모크")을 쓰면 테스트 케이스 섹션에 `[SMOKE:data-access]` 케이스 필수 — mock만으로는 커버리지로 인정하지 않으며 누락 시 plan 미완. 레이어별 설계 각 섹션에는 **실제 작성될 코드**를 코드 블록으로 넣는다 — 신규 파일은 전체, 수정 파일은 변경 부분; 의사코드·시그니처만 쓰기 금지. 저장 시 validator 훅이 검증한다(설계 섹션에 코드 블록 없으면 거부). "레이어별 설계"와 "응답 코드"는 유닛별 설계 + 각 진입점의 interface contract로 읽되, 이 프로젝트가 실제로 가진 레이어·진입점·결과에 맞춰 프로젝트가 쓰는 이름 그대로 설계한다; HTTP 응답 코드가 없는 프로젝트에선 "응답 코드"를 각 진입점의 outcome contract로 읽는다.

## 4. 변경 모드
`${CLAUDE_PLUGIN_ROOT}/templates/revision.md` 기반 `.beaver/output/revision/<domain>/<feature>-revision-<YYMMDD>-<N>.md`. 원본 spec/plan은 참조만. 변경 코드 섹션에는 영향받는 파일별 **실제 코드**를 변경 전 → 후 코드 블록으로 넣는다 — plan 설계 섹션과 같은 "눈으로 바로 이해" 규칙.

## 4.5 규약 영역 보강 (새 영역일 때만)
§3 스캔으로 기능이 현 `docs/`·`CLAUDE.md`에 **없는 새 규약 영역**(websocket·payment·cache·실시간 등)을 도입하는지 판별한다.
- **기존 영역의 루틴 기능** → 묻지 않고 기존 규약을 따른다.
- **새 영역** → plan 확정 후 **"이 설계의 규약을 docs에 반영할까?"** 제안 — 무엇을 쓸지 미리보기(`docs/<topic>.md` 신규 + `CLAUDE.md` 섹션·체크리스트 링크). 승인 시 `analyze`와 같은 doc 구조로 작성, 생략 시 plan 문서에만 남긴다.
- 코드가 나오기 전 작성이므로 머리에 **draft 마커**를 단다: `<!-- beaver:draft 기획 기준 · ship 전 미확정 -->`. build가 코드 따라 갱신하고 ship §1이 검증·확정(마커 제거)한다.
- 편집은 stick 브랜치에서 일어나 기능과 한 묶음으로 이동한다.
