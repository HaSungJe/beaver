# test — 자가회복 전체 회귀 (독립)

프로젝트 **전체** 테스트 스위트(`commands.test`)를 실행한다. green → 보고하고 멈춘다. **red** → **소스**를 자율 수정하고 green까지 재실행한 뒤(가드 범위 내), 수정을 커밋하고 확인 후 푸쉬한다. ship에서 분리해 필요할 때 호출; build는 테스트를 작성만 하므로 실행과 회복은 이 스킬에서 일어난다.

## 0. 전제 (깨지면 안내 후 중단)
- `.beaver/config.json` 존재 + `commands.test` 정의. 없으면 중단하고 `/beaver:analyze` 안내.
- **원격 리모트가 있는 브랜치에서 실행** — `git rev-parse --abbrev-ref --symbolic-full-name @{u}`(또는 `git ls-remote --exit-code origin <branch>`) 확인. 이는 **stick worktree**(로컬 전용 브랜치)를 의도적으로 배제한다: 자가회복은 소스를 수정·커밋·푸쉬하므로 의존성 갖춘 실제 개발자 체크아웃 — 보통 `/beaver:ship` 후 원래 브랜치 — 이 필요하다. 원격 없으면 중단하고 그리로 안내, stick 안 금지.
- 체크아웃에 의존성이 설치돼 있다고 가정한다. `commands.test`가 모듈 해석에서 실패하면 의존성을 설치(`commands.setup`/`npm ci` 등)하고 재시도 — 이 스킬 안에서 즉흥 처리하지 않는다.

## 1. 자가회복 루프
현재 체크아웃에서 `commands.test` **전체**를 실행한다. 설정된 커맨드를 그대로 쓴다(단위 테스트가 아니어도 됨 — 타입체크·빌드·다른 러너 가능). `commands.test_smoke`가 정의돼 있고 `commands.test`에 포함되지 않으면 같은 회귀의 일부로 함께 실행한다(스모크 red 도 아래 루프로 진입). 위험 매핑 스냅샷 실패는 새 매핑 민감 필드가 생겼다는 뜻(docs/testing.md "Data-Access 스모크") — 스냅샷을 맹목적으로 갱신하지 말고, 새 필드의 조건 사용처에 스모크 케이스가 있는지 먼저 확인한다(스냅샷 갱신은 테스트 수정으로 취급 → ASSESS 게이트).

```
attempt = 0 ; last_failure = null
loop:
  commands.test 실행
  green → §2로
  red:
    attempt += 1
    러너 출력 파싱 → 실패 테스트 + 에러 시그니처 식별
    가드 a — attempt > maxAttempts(기본 3): 중단, 보고(포기). §3으로.
    가드 b — 실패셋 + 에러 == last_failure: 중단, 보고(진전 없음). §3으로.
    판단 — 테스트가 명백히 틀렸나? (아래 기준)
      예 → 일시정지, 사람한테 질문: [테스트 수정 / 소스 계속 시도 / 중단]
             테스트 수정 수락 → 테스트 패치(유일하게 허용되는 테스트 수정), 루프 재개
             소스 계속       → 소스 버그로 취급, 계속
             중단            → 중단, 보고. §3으로.
      아니오 → 소스만 수정, last_failure = 현재, 루프
```

### 불변 규칙 (위반 금지)
- **테스트 파일은 read-only.** 실패를 없애려고 프로젝트 테스트 글롭에 매치되는 파일을 절대 수정하지 않는다. 테스트 글롭은 `.beaver/config.json`(`test.globs` 있으면)에서, 없으면 `**/*.test.*`, `**/*.spec.*`, `**/__tests__/**`, `**/test/**`, `**/tests/**` 폴백. 소스 수정 전 대상 경로를 이 글롭과 대조해 매치되면 거부 — 판단 게이트에서 사람이 테스트 수정을 승인한 경우 제외.
- **기본 가정은 항상: 소스가 틀림.** 테스트 탓으로 돌리는 건 쉬운 길이 아니라 어려운 길이다. 판단 게이트만 테스트를 수정할 수 있고, 그것도 사람의 명시적 승인이 있어야 한다.
- **소스 수정은 자율.** 시도 사이 비-테스트 소스 패치엔 승인 불필요.

### "테스트가 명백히 틀림" 기준 (판단)
소스가 아니라 테스트가 문제라는 구체적 증거가 있을 때만 멈추고 묻는다. 애매하면 소스 버그로 취급하고 계속 시도:
- 소스가 이미 문서화된 규약/스펙에 부합하는데 테스트 기대값이 모순되거나, OR
- 테스트가 더 이상 존재하지 않는 API/동작을 검사(stale test), OR
- 스펙에 비춰 명백히 올바른 소스 수정을 했는데도 같은 실패가 시도 내내 지속.

memory(`.beaver/memory/`)는 실패를 프로젝트 규칙에 비춰 해석할 때 읽는다.

### maxAttempts
기본 3. `.beaver/config.json`에서 조정 가능(예: `healing.maxAttempts`).

## 2. green — 커밋 & 푸쉬
- **첫판 green** (이번 실행에서 소스를 건드리지 않음): 전체 회귀 통과만 보고(러너가 주면 스위트/건수 요약). 커밋·푸쉬 없음. §2.5로.
- **자가회복 후 green** (소스를 수정함):
  - 수정을 자동 커밋(제목 = 날짜, 불릿 본문, 프로젝트 커밋 메시지 규약 따름; Co-Authored-By 없음).
  - diff를 보여주고 **푸쉬 직전 확인**("green 달성, diff 이거임 — 푸쉬?").
  - 승인 → 원격에 푸쉬. §2.5로.

## 2.5 동작 위험 점검 (green 후 — 스위트 밖 AI 탐색)
스위트는 단언한 것만 증명한다. green 후, **테스트가 못 보는 동작 위험** — 컴파일·린트·테스트를 통과하지만 런타임에 오동작할 수 있는 코드 — 를 점검한다.

**범위 (기본: 변경 표면)** — 최근 작업으로 바뀐 파일(예: `git diff origin/<기본 브랜치>...HEAD` 파일셋, 또는 마지막 ship된 기능의 커밋들) + 그 직접 의존자(임포터/호출자/배선). 프로젝트 전체 스윕은 명시 요청("전체 점검") 시에만. fan-out은 표면 크기에 맞춘다(병렬 우선: Workflow parallel / Task 분배 / 순차 폴백), 각 에이전트에 명시적 읽기 범위 지정.

**점검 관점** (동작 위험만, 스타일 금지):
- **조용한 실패 경로** — 삼켜진/포괄 catch, await 누락 async, 에러 처리 없는 fire-and-forget, 전파 누락.
- **배선 누락** — 작성됐지만 등록/구독/라우팅 안 된 유닛(DI 등록, 이벤트 리스너, 라우트, 스케줄러); 테스트 mock이 이를 가린다.
- **계약 드리프트** — mock 경계가 가리는 계층 간 불일치: 스키마 vs 데이터 형태, 참조되지만 정의 안 된 env/설정, 형제 진입점 간 응답 형태 불일치.
- **상태·동시성** — 다중 쓰기 흐름의 트랜잭션 경계 누락, 비멱등 재시도, check-then-act 레이스.
- **경계 동작** — null/빈값/0 경로, 페이지네이션 한계, 타임존/인코딩, 리소스 정리(커넥션·핸들).
- **테스트 사각지대** — 스위트가 mock으로만 지나가는 동작(docs/testing.md mock 경계 규칙): 표시만, 추측 금지.

**검증 규율** — 모든 발견은 보고 전에 실제 코드로 검증한다: 실제 경로를 읽고 근거(`경로:라인`)를 인용하고 구체적 실패 시나리오(입력/상태 → 오동작)를 명시. 런타임에 오동작할 수 없으면 발견이 아니다.

**산출과 처리 (보고 전용 — 자동 수정 금지)** — 자가회복 루프는 red 테스트만 고친다; 점검 발견은 사람이 판단할 가설이다:
- `.beaver/output/inspection/<YYMMDD>-inspection.md` 작성(심각도 순: 위험 / 근거 경로:라인 / 실패 시나리오 / 처리 제안) + 채팅 요약.
- 수정은 `/beaver:fast`(또는 plan) → `build` 경로로; 사소한 수정만 사용자 명시 승인 하에 인라인 적용 후 §1 재실행.
- 발견 0건 → 점검한 범위와 함께 명시적으로 보고 — 지어내지 않는 정직이 우선.

## 3. red — 보고
루프가 포기(가드 a/b)하거나 사람이 중단했을 때: 어떤 테스트가 여전히 실패하는지 정확히 보고(러너 출력 인용)하고 무엇을 시도했는지 알린다. 폴백 수정 경로는 여전히 해당 기능의 `/beaver:plan`→`/beaver:build`, 다시 ship 후 `/beaver:test` 재실행.
